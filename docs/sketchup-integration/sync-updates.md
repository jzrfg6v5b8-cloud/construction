# 同步与版本更新

## 身份与基线

模型属性字典保存 `projectId`、`schemeId`、已应用 revision、源对象稳定 ID、源字段摘要和最后同步 operation ID。同步开始前计算当前模型摘要，并与上次回执和新包基线比较。

版本不是简单字符串大小比较：`WIP`、`R`、`A` 是发布状态，序号在同一状态内递增。跨方案、跨项目或降级必须阻断并走“另存副本/显式回滚”。

## 更新流程

1. 保存 `.skp` 并创建带时间戳的只读备份；
2. 校验包、资源哈希、schema 和兼容矩阵；
3. 以稳定 ID 计算新增/更新/建议删除/冲突；
4. 展示几何、变换、SKU、房间、材质和场景差异；
5. 用户确认后在单 operation 应用；
6. 重新统计并输出回执、handoff manifest 和未解决项。

重复应用同一输入哈希应返回“无变化”。中断或异常必须 `abort_operation`；不得把半应用状态写成成功 revision。

## 冲突策略

- 源端变化、本地未变：可更新；
- 源端未变、本地变化：保留并标记本地偏离；
- 双方变化：阻断该对象，人工选择；
- 源端删除、本地变化：默认保留到“待删除”标签；
- 组件库版本变化：先生成替换预览，禁止按显示名静默替换。

尺寸、SKU、承重/固定构件、已关联 LayOut 标注的对象属于高风险变化，即使无技术冲突也需人工确认。

## LayOut 后续

每次 `.skp` 发生可见变化，旧 LayOut/PDF 都应标记过期。插件只能把受影响场景、页面建议和检查项写入 manifest。随后仍由用户完成：

1. 打开 `.layout` 并刷新模型引用；
2. 检查场景、比例及全部关联尺寸；
3. 点击导出 PDF。

不得因 SketchUp 同步成功就把 LayOut 或 PDF 标为最新。
