# 验收演示

使用脱敏的 A03023 副本。演示前记录 SketchUp/LayOut、插件、schema 和组件库版本；不得把仓库中的占位哈希或 Demo 数量当作真实验收结果。

## 演示步骤与通过条件

1. **安装**：从扩展管理器安装经校验 `.rbz`；版本和支持矩阵可见。
2. **非法包**：导入含路径穿越或错误哈希的 fixture；预检阻断且模型无变化。
3. **合法预检**：选择 A03023 JSON；显示项目/方案/revision、单位、新增/更新/冲突和资源摘要。
4. **应用与撤销**：确认后一次 operation 写入；撤销一次可完整恢复，再重做/重新导入保持幂等。
5. **模型核查**：抽查坐标、尺寸、朝向、标签、材质、稳定 ID 和指定场景；固定比例场景为正投影。
6. **组件/SKU**：重算唯一 SKU、实例、房间及未绑定数；与模型人工抽样和 manifest 一致。示例的一个未绑定项必须阻断批准状态。
7. **同步**：修改一个源对象并本地修改同一对象；下一次预检显示双向冲突，不静默覆盖。
8. **桥接**：错误 Origin、过期令牌、重放 nonce 和超限正文均被拒绝；合法写请求仍需 SketchUp 内确认。
9. **恢复**：模拟应用异常；operation 中止，revision 不推进，备份可恢复，日志无令牌/完整客户数据。
10. **交接**：保存 `.skp`，生成包含实际哈希、场景、SKU 和 blocking 项的 handoff manifest。

## LayOut 三步演示

验收人使用团队自行创建的 `.layout`（仓库不提供二进制文件）：

1. 打开指定 `.layout`，在文档引用中刷新 manifest 对应 `.skp`；
2. 检查每页场景、投影、比例、视口裁切和所有关联尺寸，并签署检查清单；
3. 用户点击 PDF 导出，核对页数、修订号、水印、字体和文件名。

通过标准是“人工动作被收敛到上述三步且问题可见”，不是“SketchUp Ruby 自动控制 LayOut”。Ruby API 不能直接完成 LayOut 全部刷新、尺寸复核和导出；演示不得预录或脚本化点击来声称自动 PDF。

## 验收证据

保存脱敏的输入哈希、预检报告、操作回执、模型/组件库摘要、SKU 对账、handoff manifest、LayOut 人工签署清单和导出 PDF 哈希。任何 blocking 项未关闭时只能标记 DRAFT。
