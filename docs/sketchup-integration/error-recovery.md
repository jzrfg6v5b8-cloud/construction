# 错误恢复

## 首要原则

发生错误先停止写入，保留原文件和诊断信息，不在唯一项目文件上反复尝试。错误恢复以“最近已知良好 `.skp` + 输入包 + 回执/manifest”为基线。

## 导入失败

1. 若尚未应用，仅修正 JSON/资源后重新预检；
2. 若 operation 抛错，插件应 `abort_operation`；确认模型 revision 未被推进；
3. 若 SketchUp 仍响应，可使用一次撤销恢复整个 operation；
4. 若状态不确定，关闭且不保存，重新打开最近备份；先在副本复现；
5. 比较输入哈希、模型摘要和回执，确认后再重试。

不要通过编辑 manifest 把失败伪装成成功。缺资源、哈希不符和版本不兼容不能靠“忽略并继续”恢复。

## 中断或崩溃

重启后插件只读取恢复标记，不自动续写。若发现“开始但未提交”的 operation ID，将模型标为 `RECOVERY_REQUIRED`，要求用户选择最近备份或人工审查。诊断包仅包含脱敏日志、版本、operation ID、错误码和计数；发送前人工检查隐私。

## 同步冲突

对冲突对象复制当前模型到隔离组，分别保留源端和本地版本，禁止自动平均尺寸或合并变换。解决后产生新 revision 和决策记录，不覆盖原输入包。

## LayOut 问题

- 引用丢失：停止导出，按 manifest 哈希定位正确 `.skp`，不要仅按同名文件重连；
- 关联尺寸异常：撤回到正确模型引用，逐条重新吸附和复核；
- 导出失败：先另存 `.layout` 副本，检查字体、磁盘空间、页面和渲染模式，再由用户重新点击导出；
- PDF 已发出但模型更新：将 PDF 标记过期并发布新修订，不覆盖旧发布记录。

SketchUp 插件无法替用户完成 LayOut 的恢复、刷新、尺寸判断或 PDF 导出。

## 升级回滚

保留旧插件安装包、组件库快照和兼容说明。新版迁移失败时禁用新版，恢复模型副本与旧组件库，在旧版 SketchUp/插件组合中打开；不要用新版另存后的唯一文件直接降级。
